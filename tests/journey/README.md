# journey — 一条端到端主线(本地形态,无 HTTP)

不追求穷尽,而是像真实使用那样把**典型功能连成一个故事**跑一遍,验证它们协作
无碍。每个功能的边界与负路径仍在各自目录细测(`schema` / `search` /
`time_machine` / `readonly_guard` / `server` …);这里只走**正路径大巡游**。

一个测试 `test_full_local_journey`,顺序讲完整故事:

1. **建库** — 声明 `notes` 表(`title`/`body` 两列可搜索)。
2. **混合时间批量写入** — 30 条中文技术笔记(缓存/终端/向量检索/存储/全文
   检索/机器学习 6 主题各 5 条):
   - 批 A、B 用冻结时钟落到**历史 ds**(`20260101` / `20260104`);
   - 批 C `undo` 回**真实当前时间**写入(“今天”的数据)。
   写入全程走 `db.insert` / `db.delete` 公共路径。
3. **结构化查询** — 计数 / `topic` 过滤 / `ORDER BY … LIMIT` / 参数化 `IN`。
4. **中文语义检索** — `search(body, '缓存淘汰策略')` 等把想要的那条捞到最前;
   并演示 `search()` 与结构化条件同句组合。
5. **时光机回溯** — 按历史 `ds_end` 看到当时的世界;含**回溯已删除的历史行**
   (n03 于 `20260104` 删除:as-of `20260103` 仍可检索,as-of now 检索不到)。
6. **持久化** — 关库重开,数据与 search 仍在。
7. **rebuild** — 清派生索引、重放文件镜像:计数不变、历史 ds 保真、search 复活。
8. **写一次性** — 重插既有主键被拒。

## 时间维度怎么造

公共写入永远盖“今天”的 `ds`。为让时光机真能跨天回溯,测试 monkeypatch 写入路径
的时钟(`seekbase.service.write_service.today/now`)把前两批钉到历史 `ds`,第三批
`undo` 回真实时间——**只改时钟,不碰引擎内部**,写入路径本身不变。
